# Contributing

## Бранчи и PR

### Основные ветки

- `main` / `master` (release) — продакшн-ветка, стабильная версия.
- `develop` (integration) — ветка для интеграции и тестирования.

### Процесс слияния

**⚠️ КРИТИЧЕСКИ ВАЖНО: Слияние в `main`/`master`**

**В `main`/`master` ни в коем случае ничего не сливаем без разрешения лида.**

**Процесс:**

1. **PR в `develop`**: Открывай PR в `develop` (или целевую ветку по задаче). CI должен пройти (lint + build).
2. **Полное тестирование в `develop`**: После слияния в `develop` проводится полное тестирование функциональности.
3. **Разрешение лида**: Только после успешного тестирования в `develop` лид даёт разрешение на слияние в `main`/`master`.
4. **Повторное тестирование**: После слияния в `main`/`master` проводится повторное тестирование для подтверждения стабильности.

**Исключения:**

- Только лид может создавать PR в `main`/`master`.
- В экстренных случаях (критические баги в продакшене) процесс может быть ускорен, но всё равно требует разрешения лида.

### Название веток

- **Формат:** `SPH-1`, `SPH-123`, `HOTFIX` или другой формат с дефисом и цифрами.
- **Запрещено:** Ветки не должны начинаться с `feat/`, `fix/`, `chore/` и т.д. (проверяется автоматически).
- **Разрешено:** Ветки `HOTFIX` (в любом регистре) разрешены для срочных исправлений.

## Чеклист перед PR

- `pnpm lint`
- `pnpm build`
- Обновлённые env/конфиги описаны в README или docs.
- Нет временных заглушек/console.log (warn/error допустимы при необходимости).
- Имена коммитов понятны; можно использовать conventional commits (`feat: ...`, `chore: ...`) — не обязательно, но желательно.

## Кодстайл

Проект использует **Feature-Sliced Design (FSD)** для организации кода. Подробности см. в `docs/ARCHITECTURE.md`.

### Структура слоёв

- **`src/app`** — маршруты App Router, только композиция виджетов, минимум логики.
- **`src/shared`** — инфраструктура: утилиты, стили, конфиги, store core. Без бизнес-логики.
- **`src/domains`** — доменные модули: модели, репозитории, сервисы, сторы MobX. Каждый домен изолирован.
- **`src/widgets`** — UI-блоки: сборка доменных данных в переиспользуемые компоненты.

### Правила

- Импорты только через алиасы `@/...`.
- Строжайший TS включён: не ослабляем без причины.
- Бизнес-логика — только в `src/domains`, UI — в `src/widgets`, страницы — только композиция.
- MobX: сторы создаём в RootStore, регистрируем в конструкторе. Не используем синглтоны.
- Клиентские компоненты (`"use client"`) только там, где нужен state/effects.
- Домены не импортируют друг друга напрямую (используй кросс-доменные сервисы в shared при необходимости).
- Страницы не должны напрямую импортировать домены — используй виджеты.

### Порядок импортов

```typescript
// 1. Shared (инфраструктура)
import { useStores } from "@/shared/store";

// 2. Domains (бизнес-логика)
import type { User } from "@/domains/user/model";

// 3. Widgets (UI-блоки)
import { UserProfile } from "@/widgets/user-profile/user-profile";
```

## Коммиты

**Обычные коммиты:**

- Используйте `pnpm commit` вместо `git commit` для создания структурированных коммитов.
- **Формат коммита:** `SPH-1 feat: описание` — название ветки автоматически добавляется в начало коммита.
- При использовании `pnpm commit`:
  1. Выберите тип изменения (feat, fix, chore и т.д.)
  2. Введите описание коммита
  3. Название ветки (например, `SPH-1`) автоматически добавится в начало
- **Автоматические проверки перед коммитом:**
  - ✅ Форматирование и линтинг изменённых файлов (lint-staged)
  - ✅ Полная проверка линтинга всего проекта
  - ✅ Проверка типов TypeScript
  - ✅ Проверка сборки приложения
  - ✅ Проверка названия ветки
- **Ошибок быть не должно** — все проверки должны проходить успешно. Коммит будет заблокирован при наличии ошибок.

**Срочные коммиты (редкое исключение):**

- Для крайне срочных случаев можно использовать `git commit -m 'Test' -n` (флаг `-n` пропускает hooks).
- **Используйте только в редких исключительных случаях**, когда действительно нужно быстро закоммитить без проверок.
- В обычной работе такие коммиты не должны использоваться — все ошибки должны быть исправлены перед коммитом.

## Hooks и инструменты

- Husky: `pnpm prepare` после `pnpm install` для установки git hooks.
- Lint-staged гоняет `eslint --fix` и `prettier --write` на стейдженных файлах.
- Ветки в формате `feat/`, `fix/`, `chore/` и т.д. блокируются автоматически (кроме HOTFIX).
