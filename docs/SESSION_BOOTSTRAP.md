## 1. Обязательный гейт перед стартом задач

До любых изменений в коде агент обязан:

1. Прочитать **весь обязательный пакет** документации (см. §2 ниже).
2. Свериться с `cursor/ARCHITECTURE.md` и выбрать профиль выполнения задачи: через агента и/или релевантные скиллы.
3. Сформировать краткое резюме: общая картина, текущее состояние, выбранная следующая задача.
4. Задать пользователю контрольный вопрос:
   - **"Я прочитал весь обязательный контекст и готов приступить к TASK-XXX. Могу ли приступать?"**
5. Ждать явного подтверждения пользователя.

Без подтверждения пользователя к реализации **не приступать**.

## 3. Сжатый контекст продукта (нельзя забывать)

- **Продукт:** SFERA LMS — курсы для кандидатов по ссылке, отклики, ведение кандидатов в пространстве компании.
- **Multi-tenant:** **Пространство** изолировано; пользователь **в одном** пространстве; **подписка** активна или приостановлена — при приостановке **все** (внутренние + кандидаты) **не работают** с продуктом как обычно.
- **Роли:** суперадмин (пространства/подписки) → админ компании (сущности) → HR (просмотр, **статусы** кандидатов, без правки сущностей) → кандидат (по ссылке, без полноценного ЛК в MVP).
- **Жёсткие правила:** курс для кандидата по **уникальной ссылке**; **один отклик = один курс**; кандидат может проходить **несколько** курсов; первый урок-**анкета** при создании курса (продуктовый контекст).
- **MVP нет:** интеграций HRM/ATS, сложной аналитики, mobile-приложения, i18n, ЛК кандидата.

## 4. Архитектура кода (FSD + стек)

- **Стек:** Next.js 16 App Router, React 19, TypeScript strict (+ `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`), MobX, Zod (`src/shared/config/env.ts`), SCSS modules, pnpm 10, Node 20.
- **Слои:** `shared → domains → widgets → app`. Страницы **не** тянут домены напрямую — только **widgets**. Домены **не** импортируют друг друга. Бизнес-логика **не** в `shared`.
- **Состояние:** `RootStore` + провайдер; новые сторы регистрировать в `src/shared/store/root-store.ts`; без глобальных singleton.
- **Импорты:** алиас `@/*` от `src/`.
- **Env:** `getPublicEnv()` в `layout`; нужны `NEXT_PUBLIC_APP_ENV`, `NEXT_PUBLIC_API_URL`; иначе падение на старте/сборке.
- **Точки входа:** `src/app/layout.tsx`, `src/app/providers.tsx`, маршруты под `src/app/...`.

## 5. Задачи и прогресс (как работать поэтапно)

- Источник правды по объёму работ: **`docs/tasks.json`**.
- Лог между сессиями: **`docs/progress.md`** (после задачи — краткий summary: что сделано, что проверено, `TASK-xxx`).
- **Одна задача за сессию** где возможно; не менять код вне задачи.
- Перед взятием задачи: все `dependencies` в JSON должны быть `done`.
- Завершение: пройти **все** `test_steps` из задачи; затем `status: "done"` **только** если проверки пройдены; не удалять и не переписывать задачи в JSON ради «удобства».

## 6. Алгоритм старта сессии (чеклист)

1. Прочитать полный пакет документов из §2.
2. Определить текущую задачу из §0 (или выбрать `pending` с наивысшим приоритетом и готовыми зависимостями).
3. Проверить `dependencies`: все должны быть `done`.
4. Сформировать короткий отчёт пользователю: общая картина, что уже сделано, что планируется следующим шагом.
5. Явно спросить разрешение: **"Могу ли приступать?"**
6. Только после подтверждения — выполнять изменения по задаче.

---

## 7. Алгоритм конца сессии

1. Выполнить `test_steps` задачи (или зафиксировать блокер в `progress.md`).
2. Если задача закрыта: выставить `status: "done"` в `docs/tasks.json` для соответствующего `id`.
3. Дописать `docs/progress.md`: `TASK-xxx`, итог, команды проверки (`pnpm lint`, `type-check`, `build` — если уместно).
4. Обновить **§0** этого файла: следующая задача / следующий шаг / ветка.
